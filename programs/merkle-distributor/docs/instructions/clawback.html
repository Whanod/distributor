Clawback Instruction

The clawback instruction allows the admin of a Merkle Distributor to reclaim any unclaimed tokens after a specified clawback period has elapsed.

Overview

This instruction enables the admin to reclaim tokens that haven't been claimed after the clawback start timestamp has passed. This serves as a safety mechanism to recover tokens that may never be claimed, preventing them from being locked forever in the distributor.

Parameters

This instruction doesn't take any parameters directly, as all necessary information is derived from the provided accounts.

Accounts Required

  1. distributor (write): The Merkle Distributor account.
  2. from (write): The token vault containing the tokens to recover.
  3. to (write): The token account designated to receive the clawed back tokens.
  4. admin (signer): The admin of the distributor who has authority to execute the clawback.
  5. token_program (read): The SPL Token program.

Validation Checks

The instruction performs the following validations:

  1. Ensures the current timestamp is at least equal to the clawback start timestamp.
  2. Verifies the admin is the signer.
  3. Confirms the distributor has not already been clawed back.
  4. Checks that the destination account is the designated clawback receiver.

Clawback Process

If all validations pass, the instruction:

  1. Calculates the amount to clawback: max_total_claim - total_amount_claimed
  2. Marks the distributor as clawed back to prevent further claims.
  3. Transfers the calculated amount from the token vault to the clawback receiver.

Security Considerations

  • Once a distributor is clawed back, all future claims will be rejected.
  • The clawback start timestamp must be set when creating the distributor and must be at least one day after the end of the vesting period.
  • Only the designated admin can execute the clawback.
  • The tokens can only be transferred to the designated clawback receiver account.

Notes

  • This instruction is useful for recovering tokens in cases where:
    • Users never claim their tokens
    • The Merkle tree contained errors
    • The distribution needs to be revised or canceled
  • After clawback, the distributor can still exist but will reject any new claims or locked claims.
  • The clawback receiver must be initialized with the same mint as the distributor.
